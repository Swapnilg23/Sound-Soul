Build a production-ready MVP web app called Sound2Soul.

Sound2Soul is the trust-first identity, publishing, and fan-growth platform for AI-assisted music creators.

Core positioning:
“Build your AI music identity, publish emotionally rich tracks, prove trust, collect fans, and monetize your sound.”

This is NOT a generic SoundCloud clone.
This is NOT an AI music generator.
This is NOT a licensing marketplace in MVP.
This is NOT a royalty or payout platform in MVP.

Sound2Soul exists to help AI-assisted music creators:
1. Create a trusted public music identity
2. Publish tracks with emotional context
3. Disclose AI involvement transparently
4. Self-certify basic rights information
5. Collect fans and email subscribers
6. Prepare for future monetization safely

The product should feel premium, safe, emotional, creator-first, and investor/acquirer-demo ready.

Primary MVP thesis:
AI music generation tools help people create songs. Sound2Soul helps those songs become trusted, discoverable, emotionally meaningful creator assets.

Important legal/safety principle:
Do not claim that Sound2Soul verifies copyright ownership.
Do not claim that uploaded tracks are copyright-safe, legally cleared, royalty-free, or guaranteed for commercial use.
Use cautious language:
- “AI process disclosed”
- “Creator-certified rights”
- “Human contribution details”
- “This information is provided by the creator.”
- “Sound2Soul does not provide legal clearance or copyright verification.”
- “Tracks may be reviewed or removed if rights concerns are reported.”

Avoid these phrases:
- “Copyright safe”
- “Legally cleared”
- “Verified ownership”
- “Guaranteed commercial use”
- “Royalty-free”
- “Officially cleared”

Build as a responsive web app first.
It must work beautifully on desktop and mobile browsers.
Do not build a native mobile app.

Database schema:

users:
- id uuid primary key
- email text
- role text check role in ('creator','listener','admin')
- created_at timestamp

creator_profiles:
- id uuid primary key
- user_id uuid references users.id
- artist_name text
- slug text unique
- bio text
- avatar_url text
- banner_url text
- genres text[]
- mood_identity_tags text[]
- ai_tools_used text[]
- social_links jsonb
- creator_statement text
- created_at timestamp
- updated_at timestamp

tracks:
- id uuid primary key
- creator_id uuid references creator_profiles.id
- title text
- slug text unique
- description text
- audio_url text
- cover_image_url text
- genre text
- mood_tags text[]
- soul_story text
- ai_involvement_type text
- human_contribution_checklist jsonb
- rights_confirmation jsonb
- visibility text check visibility in ('draft','public','unlisted')
- moderation_status text check moderation_status in ('pending','approved','rejected')
- is_featured boolean default false
- play_count integer default 0
- like_count integer default 0
- save_count integer default 0
- report_count integer default 0
- created_at timestamp
- updated_at timestamp

likes:
- id uuid primary key
- user_id uuid references users.id
- track_id uuid references tracks.id
- created_at timestamp
- unique(user_id, track_id)

saves:
- id uuid primary key
- user_id uuid references users.id
- track_id uuid references tracks.id
- created_at timestamp
- unique(user_id, track_id)

follows:
- id uuid primary key
- follower_user_id uuid references users.id
- creator_id uuid references creator_profiles.id
- created_at timestamp
- unique(follower_user_id, creator_id)

fan_emails:
- id uuid primary key
- creator_id uuid references creator_profiles.id
- track_id uuid nullable references tracks.id
- email text
- source text
- consent boolean
- created_at timestamp

reports:
- id uuid primary key
- track_id uuid references tracks.id
- reporter_email text nullable
- reason text
- details text
- status text check status in ('open','reviewed','resolved')
- created_at timestamp

pro_waitlist:
- id uuid primary key
- user_id uuid nullable references users.id
- email text
- desired_plan text
- created_at timestamp

Required business logic:
- Free creators can publish up to 3 public approved/pending tracks.
- Creators can create draft tracks beyond the public limit, but cannot submit more than 3 public tracks on Free.
- Tracks must complete rights confirmation before public submission.
- Public tracks default to moderation_status = pending.
- Only approved public tracks appear on Explore.
- Unlisted tracks can be accessed by URL but do not show in Explore.
- Draft tracks are only visible to the creator.
- Fan email forms must require consent checkbox.
- Report button creates a report record.
- Admin can approve, reject, feature, unfeature, or remove tracks.
- No download buttons in MVP.
- No creator payouts in MVP.
- No licensing marketplace in MVP.
- No AI generation in MVP.
- No autoplay.
- Increment play count when audio playback starts.

Supabase Row Level Security:
- Public users can view approved public tracks.
- Public users can view public creator profiles.
- Creators can only edit their own profiles.
- Creators can only create/edit/delete their own tracks.
- Listeners can only manage their own likes, saves, and follows.
- Admin can manage all tracks, reports, and featured status.
- Fan email creation should be allowed from public forms, but reading fan emails should be limited to the relevant creator and admins.
